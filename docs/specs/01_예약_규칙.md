# 01. 예약 규칙

본 문서는 도수치료 예약 프로그램의 **예약 생성·수정** 시 지켜야 할 규칙을 정의한다.
관련 코드: [app/routers/api.py](../../app/routers/api.py) `POST /api/appointments` (라인 1435 부근), `PUT /api/appointments/{aid}`.
관련 모델: [app/models/models.py](../../app/models/models.py) `Appointment`, `Treatment`.
관련 테스트: [tests/test_appointment_rules.py](../../tests/test_appointment_rules.py).

---

## 1. 중복 예약 차단

### 핵심 규칙

**같은 치료사 · 같은 시간(start_at) 의 예약 중, 치료항목(`treatment_codes`) 에 도수치료(`manual30` 또는 `manual60`) 가 포함된 예약은 한 슬롯에 하나만 허용한다.**

- 도수치료가 포함된 신규 예약 → 같은 치료사·같은 시간에 이미 도수치료 예약이 있으면 차단.
- 도수치료가 포함되지 않은 예약 (`eswt`, `injection`, `cartilage` 만) → 같은 시간에 다른 예약이 있어도 허용.

### 표현

| 기존 예약 (treatment_codes) | 신규 예약 (treatment_codes) | 같은 치료사·시간에서 |
|------|------|-------|
| `["manual30"]` | `["manual30"]` | ❌ 차단 |
| `["manual60"]` | `["manual30"]` | ❌ 차단 |
| `["manual30", "eswt"]` | `["manual60"]` | ❌ 차단 (양쪽 모두 도수 포함) |
| `["eswt"]` | `["manual30"]` | ❌ 차단 (신규에 도수 포함) |
| `["manual30"]` | `["eswt"]` | ❌ 차단 (기존에 도수 포함) |
| `["eswt"]` | `["eswt"]` | ✅ 허용 |
| `["injection"]` | `["eswt"]` | ✅ 허용 |
| `["cartilage"]` | `["injection"]` | ✅ 허용 |

> 즉 "기존 예약 또는 신규 예약 중 하나라도 도수치료를 포함하면 차단".

### 도수치료 코드 목록

현재 시드 기준 도수치료 = `{"manual30", "manual60"}`.
향후 `manual90` 등이 추가되면 `Treatment.role == "therapist"` 이고 코드가 `manual` 로 시작하는 항목 전체를 도수치료로 본다 (구체 구현 시 결정).

### 취소된 예약은 제외

- `Appointment.status == "canceled"` 인 예약은 중복 판단에서 제외한다.
- 즉 어떤 슬롯의 기존 예약이 취소되었으면 그 자리에 새 예약 가능.

## 2. 필수값 검증

다음 필드는 누락 시 예약 생성 실패해야 한다.
- `patient_id` — 필수
- `start_at` — 필수
- `treatment_codes` — 비어 있으면 안 됨

현재 구현 ([app/routers/api.py:1440-1441](../../app/routers/api.py:1440)) 은 `treatment_codes` 비어있음만 명시 검증. 나머지는 Pydantic 422 으로 처리.

---

## 3. 외래키 / 존재 검증

- 존재하지 않는 `patient_id` 로 예약 생성 시 실패해야 한다.
- 존재하지 않는 `therapist_id` 로 예약 생성 시 실패해야 한다 (또는 NULL 허용 — 현재 모델은 nullable).
- 존재하지 않는 또는 비활성(`active=False`) `treatment_codes` 가 포함되면 실패해야 한다.

---

## 4. 점심시간 차단

`SystemSetting.lunch_enabled == True` 인 경우, `lunch_start` ~ `lunch_end` 사이에 시작하는 예약은 차단.
구현: [app/routers/api.py:63-83](../../app/routers/api.py:63) `_check_lunch_block` (이미 동작 중).

## 5. 백엔드 강제 검증

위 모든 규칙은 **프론트뿐 아니라 백엔드 API 에서도 강제되어야 한다**.

> ⚠️ 현재 백엔드 구현 상태:
> - ✅ 점심시간 차단: 구현됨
> - ✅ `treatment_codes` 비어있음 검증: 구현됨
> - ❌ 같은 치료사·시간 도수치료 중복 차단: **미구현** (테스트는 `xfail` 로 표시)
> - ❌ 외래키 / 비활성 항목 검증: **부분 구현**
>
> 미구현 항목은 향후 백엔드에 추가될 것이며, 추가되는 즉시 해당 테스트의 `@pytest.mark.xfail` 마커를 제거할 것.

## 6. 에러 메시지

검증 실패 시 가능한 한 한국어로 원인을 알려줘야 한다. 예:

```python
raise HTTPException(400, "치료항목(treatment_codes)을 하나 이상 선택하세요.")
raise HTTPException(400, f"점심시간({ls_str}~{le_str})에는 예약을 잡을 수 없습니다.")
raise HTTPException(400, "같은 치료사의 같은 시간대에 이미 도수치료 예약이 있습니다.")  # 향후 추가
raise HTTPException(400, "해당 날짜에 종일 휴무인 치료사입니다.")                       # 향후 추가
```

FastAPI 기본 422 (Pydantic 검증 실패) 도 그대로 사용 가능 — 다만 `detail` 이 영문이라 가능하면 사전 검증으로 한국어 400 으로 변환할 것.

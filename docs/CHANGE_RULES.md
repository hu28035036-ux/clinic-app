# 도수치료 예약 — 기능 수정 절차 (CHANGE_RULES.md)

이 문서는 Claude Code 가 본 프로젝트의 코드를 수정할 때 따라야 할 절차를 정의한다.
[CLAUDE.md](../CLAUDE.md) 의 작업 규칙과 함께 읽을 것.

---

## 기능 수정 절차

1. **관련 spec 문서를 먼저 읽는다.**
   - 예약 관련 → [docs/specs/01_예약_규칙.md](specs/01_예약_규칙.md)
   - 휴무 관련 → [docs/specs/02_치료사_휴무_규칙.md](specs/02_치료사_휴무_규칙.md)
   - 통계 관련 → [docs/specs/03_완료카운트_통계_규칙.md](specs/03_완료카운트_통계_규칙.md)

2. **수정 대상 파일을 먼저 파악한다.**
   - 어떤 파일이 어떤 기능을 담당하는지 `Read` / `Grep` 으로 먼저 확인한다.
   - 사용자에게 "이 파일에서 이런 흐름으로 동작합니다" 한국어로 요약 후 진행한다.

3. **변경 범위를 최소화한다.**
   - 요청받지 않은 파일은 수정하지 않는다.
   - "겸사겸사 정리" 금지.
   - 기능 수정과 디자인 수정을 한 번에 섞지 않는다.

4. **테스트용 DB 에서 검증한다.**
   - `run_check.bat` 또는 `run_tests.bat` 으로 항상 임시 DB 에서 검증.
   - 운영 DB (`%APPDATA%\도수치료예약\clinic.db`) 를 절대 수동 조작하지 않는다.

5. **pytest 실행:** `venv\Scripts\python.exe -m pytest tests -v`
   - 정방향 테스트가 모두 PASS 인지 확인.
   - XFAIL 은 미구현 차단 로직(spec 에 정의되었지만 코드에는 아직 없음) — 통과로 간주한다.
   - XPASS 가 보이면 → 백엔드에 차단 코드가 추가된 것. 해당 테스트의 `xfail` 마커 제거 시점.

6. **ruff 실행:** `venv\Scripts\python.exe -m ruff check app tests`
   - `tests/` 와 `scripts/` 의 새 코드만 실제 검사 대상 (`app/**` 은 의도적으로 면제).
   - `--fix` 자동 수정 가능.

7. **`run_check.bat` 실행** — 위 둘 + DB 경로 안전 검사를 한 방에.

8. **실패 시 원인을 한국어로 기록**한다. 단순 "테스트가 실패했습니다" 가 아니라 "어떤 spec 의 어떤 규칙이 깨졌고, 어떤 코드 변경이 원인인지" 까지 적는다.

## 절대 하지 말 것 (금지 사항)

### 변경 범위 관련
- 기능 수정과 디자인 수정을 한 번에 섞지 않기.
- DB 구조 변경과 UI 변경을 한 번에 섞지 않기.
- 요청받지 않은 전체 리팩토링 금지.
- 코드 스타일 통일을 위한 대규모 포맷팅 금지.
- `pyproject.toml` 의 `app/**` lint 면제 (per-file-ignores) 를 풀지 말 것.

### 데이터 안전 관련
- 실제 운영 DB 로 테스트하지 않기.
- 운영 DB 파일을 직접 SQL 로 수동 편집하지 않기 (마이그레이션 파일 추가로 처리).
- 임시 DB 와 운영 DB 의 스키마가 달라지게 변경하지 않기 — 모든 변경은 `app/migrations/m00X_*.py` 를 통해.

### 도메인 규칙 관련
- 기존 통계 기준 임의 변경 금지.
- `manual60` 을 `count_increment=2` 로 되돌리는 변경 금지 (manual60 = 1카운트 정책 유지).
- 도수치료(`manual30` / `manual60`) 외 항목까지 중복 예약 차단을 확장하지 말 것 (spec 01 참조).
- 휴무 차단 시 12:00 기준 분기를 다른 값으로 변경하지 말 것 (spec 02 참조).

### 인터페이스 관련
- 기존 API 경로를 변경하지 말 것 (`POST /api/appointments` 등).
- DB 컬럼명 / 테이블명 변경 금지.
- 프론트에서 막는 기능은 반드시 백엔드에서도 검증 — UI 만 막고 API 는 열어두지 말 것.

### 작업 범위 관련
- 업종 변경 / 브랜딩 변경 / 용어 변경 (예: 환자 → 고객, 치료사 → 작업자) 금지.
- 도수치료 예약 관리 프로그램 외 다른 용도로의 전환 금지.
- 기존 화면 흐름 변경 금지.

## 새 기능 추가 시 하네스 갱신 의무

- 새 비즈니스 규칙이 추가되면 → 적절한 `docs/specs/*.md` 에 규칙 추가.
- 새 핵심 동작이 추가되면 → `tests/` 에 검증 테스트 추가.
- 새 마이그레이션이 추가되면 → `app/migrations/m00X_*.py` + `dosu_clinic.spec` 의 `hiddenimports` 갱신 (CLAUDE.md 참조).

하네스를 우회하는 변경은 금지. 하네스가 수정 대상에 어울리지 않으면 사용자에게 먼저 보고할 것.

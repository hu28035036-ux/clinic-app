# AI/RAG 의사결정 기록 (ai_rag_decision_record)

> 왜 지금의 구조와 정책을 선택했는지를 기록한다. 향후 변경 검토 시 근거 비교용.
> ADR(Architecture Decision Record) 포맷. 각 항목: **결정 / 배경 / 대안 / 결과 / 재검토 시점**.

---

## ADR-001. 외부 LLM API를 기본 엔진으로 쓰지 않는다

- **결정**: 모든 AI 기능은 내부 데이터·규칙·로컬 답변 조립을 1순위로 하고, 외부 LLM은 보조로만 호출한다.
- **배경**:
  - 사용자 환경이 단독 실행형 Windows 앱(`%APPDATA%\도수치료예약\clinic.db`). 인터넷이 항상 보장되지 않는다.
  - 외부 API 비용은 사용자가 부담한다. 무분별 호출 시 토큰 비용이 폭증한다.
  - 의료/예약 데이터는 PII 포함 — 외부 전송 표면을 최소화해야 한다.
  - 외부 API 응답은 비결정적 — 핵심 업무 흐름의 신뢰성을 LLM에 의존하면 안 된다.
- **대안**:
  - LLM-first: 모든 질문을 LLM에 던지고 RAG로 보강 → 비용/신뢰성 모두 불리.
  - LLM-only: RAG 없이 LLM 단독 → 할루시네이션·도메인 지식 부족.
- **결과**: Local-first AI assistant. 외부 API는 "근거 충분 + 자연어 합성 필요" 케이스만.
- **재검토**: 사용자 피드백상 자연어 응답 비중이 올라가면 ai_assist 모드 기본값 검토.

## ADR-002. local-first 구조를 선택했다

- **결정**: AI 처리 우선순위는 (1) Safety 검사 → (2) 내부 규칙 → (3) 내부 DB → (4) 내부 매뉴얼 검색 → (5) 로컬 답변 생성 → (6) 외부 API 호출 판단 → (7) 외부 API.
- **배경**: ADR-001과 동일.
- **대안**: 단순 RAG(검색 후 항상 LLM 호출).
- **결과**: 7단계 파이프라인. 6단계에서 충족하면 7단계 LLM 호출 0회.
- **재검토**: 단계별 로그(`reason_code`)로 6단계 차단 비율 모니터링.

---

## ADR-003. API key 없어도 기본 기능이 동작해야 한다

- **결정**: API key 미설정 환경에서도 keyword RAG·내부 DB 조회·규칙 기반 응답·고정 템플릿 SMS 등 모든 로컬 기능은 정상.
- **배경**:
  - 일부 사용자는 처음에 API key 없이 도구만 사용.
  - API key 발급 전이라도 매뉴얼 Q&A의 검색 결과는 보여줘야 한다.
- **대안**: API key 없으면 모든 AI 기능 503.
- **결과**: `local_only` 모드 기본 보증. provider/embedding 차단은 명시적 reason_code로.
- **재검토**: 신규 기능 추가 시 "key 없는 환경에서 동작?" 체크리스트 통과 필수.

## ADR-004. 외부 API는 보조 기능으로만 둔다

- **결정**: 외부 LLM/Embedding은 다음 모두를 만족할 때만 호출:
  - 모드가 `local_first` 또는 `ai_assist`
  - sources ≥ 1, confidence ≥ 임계, PII 위험 없음
  - 사용자 의도가 자연어 합성/요약/문장 다듬기
- **배경**: ADR-001, ADR-003.
- **대안**: 호출 빈도 제한만 두고 나머지는 LLM 사용.
- **결과**: 7단계 게이트. 위반 시 `reason_code`로 차단.
- **재검토**: 게이트 false-negative(실제로는 LLM 호출이 더 도움됐을 케이스) 분석.

## ADR-005. SQLite 기반으로 먼저 시작한다

- **결정**: chunk/vector 영속화는 SQLite (`knowledge_chunks`, `knowledge_vectors`). 외부 vector DB 미도입.
- **배경**:
  - 단독 실행형 앱. 외부 서비스 의존성 추가는 배포·운영 부담.
  - PyInstaller onedir에 외부 vector DB(faiss/chromadb)를 동봉하면 빌드 크기 폭증.
  - knowledge 규모가 작다(수십~수백 청크). SQLite로 충분.
- **대안**: faiss / chromadb / pgvector.
- **결과**: SQLite + 단순 cosine 검색. 추후 규모 커지면 재검토.
- **재검토**: chunk 수 > 5000 또는 검색 latency > 500ms.

## ADR-006. 외부 vector DB를 초기에는 쓰지 않는다

- **결정**: ADR-005와 동일 결론. 18-5에서 SQLite 기반 vector store만 도입.
- **배경**: ADR-005.
- **대안**: chromadb 임베드 모드.
- **결과**: 의존성 추가 0. PyInstaller 빌드 영향 최소.
- **재검토**: ADR-005와 동시.

## ADR-007. Embedding은 선택 기능이다

- **결정**: 임베딩은 옵션. keyword/chunk-keyword RAG만으로도 매뉴얼 Q&A가 동작해야 한다.
- **배경**:
  - 임베딩은 외부 API 호출 비용/지연 발생.
  - 한국어 도메인 임베딩 품질이 항상 keyword 대비 우월하지 않다.
  - API key 없는 사용자 환경 보호 (ADR-003).
- **대안**: 임베딩 필수.
- **결과**: `AI_EXTERNAL_EMBEDDING_ENABLED` flag로 토글. 실패 시 keyword fallback.
- **재검토**: 평가 데이터셋(`ai_rag_quality_eval_plan.md`) 점수가 keyword < vector + 환각 증가 없음일 때.

## ADR-008. 하네스를 먼저 작성한다 (18-0)

- **결정**: 코드 구조 변경 전에 하네스/계약 테스트를 먼저 만든다.
- **배경**:
  - 기존 코드 동작을 회귀 없이 보존해야 한다 (현재 v1.3.3 응답 키 유지).
  - 하네스가 없으면 리팩터링 중 미세 깨짐을 잡을 수 없다.
- **대안**: 코드 먼저 → 테스트 보강.
- **결과**: 18-0이 모든 후속 세션의 안전망. 신규 기능마다 하네스 fixture 추가.
- **재검토**: 신규 도메인 추가 시 하네스 분리 여부.

## ADR-009. 하네스를 도메인별로 분리한다

- **결정**: `rag_harness`, `safety_harness`, `chunk_harness`, `vector_harness`, `hybrid_harness` 분리.
- **배경**:
  - 도메인 간 fixture 충돌·결합도 상승 회피.
  - 각 도메인 변경이 다른 도메인 테스트에 영향 최소화.
- **대안**: 단일 conftest에 모두 몰아넣기.
- **결과**: 도메인별 책임 명확. 컴포넌트-하네스 매트릭스(`ai_rag_test_plan.md` §6)로 매핑.
- **재검토**: 도메인 늘면 분리 단위 재검토.

## ADR-010. keyword → chunk → vector → hybrid 순서

- **결정**: 18-2 keyword 분리 → 18-3 chunker → 18-4 chunk DB → 18-5 vector → 18-6 hybrid.
- **배경**:
  - 각 단계가 이전 단계에 의존. 한 번에 모두 시도하면 장애 원인 분리 불가.
  - 단계마다 회귀 0 보장 가능.
  - 사용자 입장에서 "검색 안됨" 같은 단일 모드 회귀가 가장 큰 위험.
- **대안**: 18-3에서 chunk+vector 동시.
- **결과**: 한 세션 한 책임. Codex 검증도 단계별로 명확.
- **재검토**: 세션 진행 중 의존성 발견 시 순서 조정 가능 (단, 사용자 승인).

## ADR-011. `manual_qa.py` wrapper를 유지한다

- **결정**: `manual_qa.ask_manual_question(db, question, *, provider_override=None)` 시그니처 영구 유지.
- **배경**:
  - 라우터 / 테스트(FakeProvider 주입)가 모두 이 함수에 의존.
  - 신규 pipeline은 wrapper 내부에서 호출.
- **대안**: wrapper 제거하고 pipeline을 라우터에서 직접 호출.
- **결과**: 후방호환 보장 + 점진적 교체 가능.
- **재검토**: 모든 호출처가 새 인터페이스로 전환된 후 (현 시점에서는 영구).

## ADR-012. 기존 API 응답 구조를 유지한다

- **결정**: `/api/ai/manual/{search,ask}` 응답 키는 후방호환. 추가만 허용, 제거/이름 변경/타입 변경 금지.
- **배경**:
  - 프론트엔드(JS)와 v1.3.3 사용 중인 사용자 환경 모두 의존.
  - 응답 깨지면 즉시 사용자 화면 깨짐.
- **대안**: 새 엔드포인트 v2 추가, v1 deprecate.
- **결과**: 단일 엔드포인트 유지 + optional 필드 추가. 계약 테스트로 보호.
- **재검토**: 명시적 v2 엔드포인트가 필요해진 시점.

## ADR-013. 할루시네이션 금지를 테스트로 검증한다

- **결정**: LLM 응답이 출처와 불일치하면 차단. 의료 단정/실행 오인/출처 없는 단정 등을 정규식+규칙으로 사전 차단. 테스트로 모든 패턴 검증.
- **배경**:
  - 의료 도메인 + 예약 시스템 — 잘못된 단정은 직접 피해.
  - LLM은 비결정적 — 코드 안전망이 유일한 방어선.
- **대안**: 사용자가 답변을 검토.
- **결과**: `answer_validator` + `_RE_MEDICAL_CLAIM` / `_RE_EXECUTION_CLAIM`. 차단 시 `blocked: true`.
- **재검토**: 새 위험 표현 발견 시 패턴 추가 (테스트 동시).

## ADR-014. PyInstaller 배포 안정성을 우선한다

- **결정**: 신규 모듈은 모두 `dosu_clinic.spec` `hiddenimports`에 등록. PyInstaller 빌드 후 import 검증 테스트(18-8) 통과 필수.
- **배경**:
  - PyInstaller는 정적 분석 → lazy import는 누락되기 쉬움.
  - 빌드 통과해도 런타임 ImportError로 사용자 환경에서 깨짐.
- **대안**: dynamic discovery.
- **결과**: spec hidden import + 검증 테스트 + 의존성 추가 시점 신중.
- **재검토**: PyInstaller 대안(briefcase 등) 도입 시.

## ADR-015. requirements.txt 변경을 vector 구현 시점까지 미룬다

- **결정**: 18-0~18-4까지는 의존성 추가 0. 18-5에서만 (옵션).
- **배경**:

- **대안**: 초반에 모든 의존성 미리 추가.
- **결과**: 단계별 빌드 영향 분리. 회귀 원인 추적 단순화.
- **재검토**: 신규 hard dependency 발견 시.

## ADR-016. Feature Flag를 먼저 설계한다

- **결정**: 모든 신규 기능은 flag 뒤에서 시작. 기본 OFF.
- **배경**:
  - 사용자 환경이 다양 (저사양/오프라인). 일괄 활성화는 위험.
  - 부분 회수(rollback) 가능해야 한다.
- **대안**: 기본 ON으로 출시.
- **결과**: 11개 flag 정의(`ai_rag_architecture_plan.md` §18). 관리자 토글.
- **재검토**: flag 수가 너무 많아지면 그룹화.

## ADR-017. reindex 실패 시 기존 index를 유지한다

- **결정**: reindex 실패해도 기존 chunk/vector 보존. 부분 실패는 실패 문서만 표시.
- **배경**:
  - reindex 실패로 검색 자체가 멈추면 사용자 업무 마비.
  - 운영 중 무중단 재인덱싱 필요.
- **대안**: 실패 시 전체 인덱스 무효화.
- **결과**: `knowledge_index_runs`에 결과 기록. 메모리 retriever는 새 인덱스가 모두 commit된 후에만 교체.
- **재검토**: 실패가 잦으면 원인 분석 강화.

## ADR-018. Claude Code 자체 테스트 후 Codex 검증을 둔다

- **결정**: Claude Code의 자체 테스트 통과는 최종 완료가 아님. Codex가 독립적으로 diff·테스트 결과·로그를 검증한 후에만 다음 세션 진입.
- **배경**:
  - Claude Code는 자기 결과물을 검증할 때 편향 가능.
  - 같은 모델이 같은 가정 하에 같은 실수를 할 수 있다.
  - 외부 도구의 독립 검증이 안전망.
- **대안**: 사용자가 매번 수동 검토.
- **결과**: `latest_codex_review_request.md` + Codex 검증 게이트(`ai_codex_review_protocol.md`).
- **재검토**: Codex 검증 false-positive/false-negative 누적되면 절차 조정.

## ADR-019. 자동 수정 반복을 5회로 제한한다

- **결정**: 한 세션 안에서 같은 작업의 "테스트→실패→수정" 루프는 최대 5회.
- **배경**:
  - 6회 이상 시도하면 보통 가설이 잘못됐거나 설계가 잘못됨 (경험적 임계).
  - 무한 루프 = 토큰/시간 낭비 + 코드 점점 더 망가짐.
  - 5회 시점에 한 발 물러서서 재검토하는 게 더 빠르다.
- **대안**: 무제한 루프 / 3회 / 10회.
- **결과**: 5회 실패 시 `latest_failure_report.md` + 사용자에게 재작성 보고.
- **재검토**: 실패 패턴 분석 후 임계 조정.

---

## ADR-020. 5회 실패 시 부분 땜질이 아니라 코드 재작성으로 전환한다

- **결정**: 5회 실패 → 부분 수정 즉시 중단 → 실패 리포트 → 사용자 승인 후 코드 재작성/설계 재검토.
- **배경**:
  - 5회 실패의 패턴은 보통 "지엽적 수정으로 깊은 문제를 가리고 있음".
  - 땜질 누적은 코드 부패와 미래 회귀의 원인.
  - 처음부터 다시 작성하면 가정이 깨끗해진다.
- **대안**: 6, 7회 더 시도.
- **결과**: 실패 리포트 기반 다음 세션은 설계부터 재검토.
- **재검토**: 실패 후 재작성으로 해결되는 비율 모니터링.

## ADR-021. PII 원문은 어디에도 저장/전달하지 않는다

- **결정**: 입력/출력/로그/캐시 어디에도 PII 원문 저장 금지. sha256 해시 + 마스킹된 텍스트만.
- **배경**: 의료/환자 데이터. 법적·윤리적 책임.
- **대안**: 마스킹만 (해시 없이).
- **결과**: `pii.scan` + `ai_logging` 해시. `AiPiiBlocked` 외부 전송 직전 차단.
- **재검토**: 신규 PII 카테고리 발견 시 패턴 추가.

## ADR-022. API key는 어떤 로그에도 남기지 않는다

- **결정**: traceback 포함 어디에도 API key 부재. 관리자 화면은 "등록 여부"만 표시.
- **배경**: 키 노출은 즉시 비용/보안 사고.
- **대안**: 마스킹된 일부 노출.
- **결과**: 로그 작성 전 일관 마스킹. 예외 핸들러도 키 제거 후 raise.
- **재검토**: 신규 인증 헤더 도입 시 동일 정책 적용.

## ADR-023. 답변 문장 전체 snapshot 테스트는 금지

- **결정**: LLM 출력 전체 문자열 snapshot 금지. 키/구조/제약/카운트만 단언.
- **배경**: LLM 응답은 비결정적. 전체 snapshot은 테스트가 매번 깨짐 → 테스트 신뢰성 붕괴.
- **대안**: temperature=0 + snapshot.
- **결과**: 부분 단언 (출처 포함, 위험 표현 부재, 길이 범위, reason_code).
- **재검토**: Local Composer 결정적 출력은 부분 snapshot 허용.

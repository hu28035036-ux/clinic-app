## 2. 요구사항 덮어쓰기 규칙

AI 설계 문서는 앞으로 계속 수정될 예정입니다.

1. 사용자가 추가수정사항을 전달하면 기존 문서에 반영합니다.
2. 기존 내용과 새 내용이 겹치거나 충돌하면 **최신 사용자 지시를 우선**합니다.
3. 겹치는 내용은 중복으로 남기지 말고 최신 내용으로 덮어씁니다.
4. 덮어쓴 내용은 `AI_REQUIREMENTS_OVERRIDES.md`에 기록합니다.
5. `AI_CURRENT_DECISIONS.md`에는 현재 기준으로 확정된 최신 의사결정만 정리합니다.
6. 과거 내용은 참고만 하고, **현재 설계 기준은 `AI_CURRENT_DECISIONS.md`를 우선**합니다.

## 3. 현재 프로그램 기능 요약

현재 프로그램은 **병원 예약관리 프로그램**입니다.
(과거에 도수치료 전용으로 표현되었던 기능들은 모두 **치료항목 기반**으로 일반화하여 정리합니다.)

- 예약 수정
- 예약 취소
- 예약 상태 관리
- 캘린더 기반 예약 표시

- 시간대별 예약 표시

### 3.2 환자관리
- 환자 등록 / 검색 / 정보 수정
- 차트번호 / 이름 / 생년월일 / 연락처 관리
- 환자 메모 관리
- 예약 이력 확인

### 3.3 치료사관리
- 치료사 등록 / 수정
- 활성/비활성 관리
- 치료 가능 항목 설정

- 치료사별 예약 표시

### 3.4 의사관리
- 의사 등록 / 수정
- 활성/비활성 관리

### 3.7 완료 체크
- 예약별 완료 체크

## 4. AI 기능 전체 목표

AI 기능의 목표는 **AI가 병원 업무를 마음대로 처리하는 것이 아닙니다.**

### 4.1 목표

1. 사용자가 자연어로 명령합니다.
2. AI는 명령을 구조화합니다.
3. 프로그램은 실제 DB 기준으로 검증합니다.
4. 위험 작업은 사용자 승인 후 실행합니다.
5. 실행은 **기존 서비스 로직**을 통해 수행합니다.
6. 모든 과정은 로그로 남깁니다.
7. AI가 모르는 내용은 추측하지 않고 확인 요청 상태로 둡니다.

### 4.2 전체 구조

```
사용자 자연어 명령
  → AI 명령 구조화 (parser)
  → 프로그램이 DB 기준으로 매칭 (resolver)
  → 프로그램이 DB 기준으로 검증 (validator)
  → 후보/미리보기 표시 (preview)
  → 사용자 승인
  → 승인 직전 최종 재검증
  → 기존 서비스 로직 실행 (executor)
  → 로그 저장 (audit)
```

### 4.3 핵심 원칙

- **AI는 후보 생성 담당**
- **프로그램은 DB 검증 담당**
- **사용자는 최종 승인 담당**
- **기존 서비스 로직이 실제 실행 담당**

## 5. AI 기능 범위 (단계별)

각 기능마다 별도 항목으로 다음 **13개 필드**를 정의합니다 (`AI_CURRENT_DECISIONS.md` § 18 과 1:1 정합).

1. intent 이름
2. 사용자 명령 예시
3. 필수 입력값
4. 선택 입력값
5. DB 조회 필요 여부
6. 승인 필요 여부
7. 실행 가능 조건
8. 실행 금지 조건
9. 호출해야 할 기존 서비스 로직
10. 필요한 하네스 테스트 케이스
11. 실제 동작 확인 방법
12. 정상 케이스
13. 실패 / 예외 케이스

> **13필드가 누락된 intent 는 해당 Phase 진입 / 검증 통과 불가**.

### 5.1 1차 대상

#### create_appointment (신규 예약 생성 AI)
- intent: `create_appointment`
- 명령 예시:
  - 박환자 4월30일 9시 도수30분 예약해줘
  - 박환자 4월30일 9시 도수30 주 충 예약해줘
  - 김민수 차트번호 12345 내일 오전 10시 박치료사 도30 예약
  - 12345번 환자 5월 10일 오후 2시 김치료사 도수60 예약
  - 이영희 내일 15시 이치료사 체외충격파 예약
  - 김민수 내일 오전 10시 박치료사 ESWT 예약
- 필수: 환자 식별값(이름 또는 차트번호), 날짜, 시간, 치료항목
- 선택: 치료사, 메모, 추가 치료항목
- DB 조회: 환자 / 치료사 / 치료항목·alias / 휴무 / 기존 예약
- 승인: **반드시 필요**
- 실행 조건: 환자 1명 확정 + 치료항목 모두 확정 + 충돌 없음
- 금지 조건: 동명이인 미선택 / 치료항목 alias 충돌 / 차트번호-이름 불일치 / 과거 날짜 / 휴무 충돌 / 시간 겹침
- 호출 서비스: 기존 예약 생성 서비스 (직접 SQL 금지)
- 하네스: parser, resolver, patient-candidate, validator, approval, executor, privacy, hallucination, regression, runtime
- 실제 동작 확인: 입력 → 환자 후보 카드 / 치료항목 후보 카드 → 사용자 선택 → 최종 예약 후보 카드 표시 → 승인 → DB row 생성 → 캘린더 / 표 / 통계 즉시 반영. 승인 전 DB 변경 0건, 승인 후 정확히 1건.
- 정상 케이스: 단일 환자 / 단일 치료항목 → 승인 → 등록 성공
- 실패 케이스: 동명이인 → 선택 없으면 등록 불가; 휴무 충돌 → 등록 불가; 과거 날짜 → 경고 / 차단

### 5.2 2차 대상

#### update_appointment (예약 변경 AI)
- intent: `update_appointment`
- 명령 예시:
  - 박환자 내일 9시 예약을 10시로 변경해줘
  - 차트번호 12345 5월 10일 박치료사 → 김치료사로 변경
  - 박환자 4월30일 도수30 → 도수60으로 변경
- 필수: 대상 예약 식별값 (환자 + 날짜 + 시간 또는 예약 ID), 변경 항목
- 선택: 변경 사유 메모
- DB 조회: 대상 예약, 변경 후 시간 충돌 검사, 휴무 / 반차, 치료사·치료항목 alias
- 승인: **필수** (변경 전·후 비교 화면)
- 실행 조건: 대상 예약 1건 확정 + 변경 후 충돌 없음 + 권한 보유
- 실행 금지 조건: 대상 예약 모호 / 변경 후 시간 겹침 / 휴무 충돌 / 권한 없음 / 과거로 변경
- 호출 서비스: 기존 예약 수정 서비스 (직접 SQL 금지)
- 하네스: parser, resolver, patient-candidate, validator, approval, executor, regression, runtime
- 실제 동작 확인: 변경 전·후 비교 카드 표시 → 승인 → DB row 업데이트 → 캘린더 / 표 즉시 반영
- 정상 케이스: 단일 예약 → 시간 변경 → 충돌 없음 → 승인 → 갱신 성공
- 실패 케이스: 변경 후 같은 치료사 동일 시간대 충돌 → 차단; 휴무 충돌 → 차단

#### cancel_appointment (예약 취소 AI)
- intent: `cancel_appointment`
- 명령 예시:
  - 박환자 내일 9시 예약 취소
  - 차트번호 12345 5월 10일 예약 취소
- 필수: 대상 예약 식별값 (환자 + 날짜 + 시간 또는 예약 ID)
- 선택: 취소 사유
- DB 조회: 대상 예약
- 승인: **필수**
- 실행 조건: 대상 예약 1건 확정 + 권한 보유
- 실행 금지 조건: 대상 예약 모호 / 이미 취소된 예약 / 과거 완료 예약 / 권한 없음
- 호출 서비스: 기존 예약 취소 상태 처리 로직 (**물리 삭제 금지**, 상태값 변경)
- 하네스: parser, resolver, validator, approval, executor, regression, runtime
- 실제 동작 확인: 대상 예약 카드 표시 → 승인 → 상태값 cancelled 변경 → 캘린더 / 표 반영
- 정상 케이스: 단일 예약 → 승인 → 취소 상태로 변경
- 실패 케이스: 대상 모호 → 후보 표시·선택 요구; 이미 취소 → 안내 메시지 후 종료

### 5.3 3차 대상

#### create_leave (치료사 휴무 / 반차 등록 AI)
- intent: `create_leave`
- 명령 예시:
  - 박치료사 5월 5일 종일 휴무
  - 김치료사 다음주 월요일 오전반차
  - 이치료사 30일 오후반차
  - 박치료사 5월10일 연차 등록
- 필수: 치료사 식별값 (이름), 날짜, 휴무 유형 (종일 / 오전반차 / 오후반차)
- 선택: 메모
- DB 조회: 대상 치료사 활성 상태, 해당 날짜 기존 예약 충돌, 기존 휴무 중복
- 승인: **필수** (충돌 예약은 별도 고지)
- 실행 조건: 치료사 1명 확정 + 휴무 유형 명확 + 기존 휴무 중복 없음
- 실행 금지 조건: 치료사 모호 / 비활성 치료사 / 휴무 유형 모호 / 같은 시간대 기존 예약 다수 충돌
- 호출 서비스: 기존 휴무 / 반차 등록 서비스 (직접 SQL 금지)
- 하네스: parser, resolver, validator, approval, executor, regression, runtime
- 실제 동작 확인: 휴무 후보 카드 + 충돌 예약 목록 표시 → 승인 → 휴무 row 생성 → 캘린더에 휴무 표시
- 정상 케이스: 충돌 없는 종일 휴무 → 승인 → 등록 성공
- 실패 케이스: 같은 시간 기존 예약 충돌 → 강조 경고 + 사용자 결정 필요; 동명 치료사 다수 → 후보 선택

#### prepare_sms (예약문자 준비 AI)
- intent: `prepare_sms`
- 명령 예시:
  - 내일 예약자에게 보낼 문자 준비
  - 5월10일 예약자 문자 내용 만들어줘
- 필수: 대상 날짜 (또는 "내일")
- 선택: 치료사 / 치료항목 필터
- DB 조회: 해당 날짜 예약자 / 환자 연락처
- 승인: 불필요 (출력만, 발송 안 함)
- 실행 조건: 대상 날짜 확정 + 예약자 1명 이상
- 실행 금지 조건: **자동 발송 시도** / 외부 AI API에 환자 연락처 전체 전송 시도
- 호출 서비스: 기존 다음날 예약자 조회 + 문자 내용 생성 로직 (발송 단계 미호출)
- 하네스: parser, resolver, privacy, regression, runtime
- 실제 동작 확인: 대상 표 + 체크박스 + 붙여넣기용 텍스트 출력 → DB 변경 0건 / 발송 0건
- 정상 케이스: 내일 예약자 N명 → 체크박스·텍스트 출력 → 사용자가 직접 복사·붙여넣기
- 실패 케이스: 예약자 0명 → 안내 메시지; AI 자동 발송 시도 감지 → 즉시 차단·로그

### 5.4 4차 대상

#### summarize_today / summarize_tomorrow (오늘 / 내일 예약 요약 AI)
- intent: `summarize_today`, `summarize_tomorrow`
- 명령 예시:
  - 오늘 예약 요약해줘
  - 내일 예약 요약해줘
- 필수: 없음 (intent로 충분)
- 선택: 치료사 / 치료항목 필터
- DB 조회: 해당 날짜 예약 / 휴무 / 치료사
- 승인: 불필요 (읽기 전용)
- 실행 조건: 항상
- 실행 금지 조건: **수치 임의 생성** / DB에 없는 환자·예약 언급 / 외부 AI API에 환자 메모·연락처 전체 전송
- 호출 서비스: 기존 예약 조회 service (수정 권한 없는 read-only)
- 하네스: parser, resolver, privacy, hallucination, regression, runtime
- 실제 동작 확인: 요약 카드 표시 → DB 변경 0건; 표시된 환자 / 예약은 모두 DB에 실재함
- 정상 케이스: 오늘 N건 / 치료사별 / 시간대별 요약 표시 (DB 값 그대로 인용)
- 실패 케이스: AI가 DB에 없는 환자 / 수치를 생성하면 즉시 검증 실패 → 표시 차단

#### analyze_stats (통계 분석 AI)
- intent: `analyze_stats`
- 명령 예시:
  - 이번달 예약 통계 분석해줘
  - 박치료사 4월 완료율 알려줘
  - 5월 첫째주 시간대별 예약 분포
- 필수: 분석 기간 (월 / 주 / 일)
- 선택: 치료사 / 치료항목 / 시간대 필터
- DB 조회: 기존 통계 service 결과
- 승인: 불필요 (읽기 전용)
- 실행 조건: 기간 명확
- 실행 금지 조건: **수치 임의 생성** / 도수 30=1, 60=2 가중치 합산 방식 사용 / 치료항목별 개별 집계 방식 변경
- 호출 서비스: 기존 통계 service (read-only)
- 하네스: parser, resolver, privacy, hallucination, regression, runtime
- 실제 동작 확인: 통계 요약 카드 → 모든 수치는 DB / 기존 통계 service 결과와 일치
- 정상 케이스: 기간 / 필터 명확 → 기존 통계 service 호출 → AI는 결과를 한국어로 요약만
- 실패 케이스: AI가 새 수치 생성 시도 → 즉시 차단; 기간 모호 → `needs_clarification`

- 실행 금지 조건: **자동 수정** / 환자 자동 병합 / 환자 자동 삭제 / 외부 AI API에 환자 목록 전송

- 정상 케이스: 차트번호 중복 후보 N건 표시 → 사용자가 환자관리 화면에서 직접 검토

## 6. 신규 예약 생성 AI 설계

신규 예약 생성 AI는 1차 핵심 기능입니다.

### 6.1 추출 필드 (AI 파싱 결과)

- intent
- patient_name
- chart_number
- date_text
- time_text
- therapist_name
- treatment_text (단일 텍스트)
- treatment_items (배열, 다중 치료항목)
- memo

### 6.2 처리 흐름

1. 사용자 명령 입력
2. AI가 구조화 JSON 생성 (parser)
3. 프로그램이 환자명/차트번호로 환자 검색 (resolver)
4. 프로그램이 환자 후보의 차트번호·이름·생년월일·연락처를 **내부 DB**에서 확인
5. 프로그램이 치료사 검색
6. 프로그램이 치료항목·약어·alias 검색
7. 프로그램이 날짜·시간 해석
8. 휴무·반차·중복·시간 겹침 검증 (validator)
9. 예약 후보 표시 (preview)
10. 사용자 승인
11. 승인 직전 **최종 재검증**
12. 기존 예약 생성 로직으로 저장 (executor)
13. 로그 기록 (audit)

> **승인 전에는 절대 예약을 저장하면 안 됩니다.**

### 6.3 중요 원칙

1. `treatment_text`는 도수치료 전용이 아니라 DB에 등록된 치료항목과 매칭해야 합니다.
2. "도수30 주 충"처럼 여러 치료항목·약어가 함께 입력될 수 있으므로 `treatment_text` 단일값뿐 아니라 `treatment_items` **배열 구조**도 고려해야 합니다.
3. `treatment_items`는 여러 치료항목 후보를 담을 수 있는 구조여야 합니다.
4. 각 치료항목은 DB 치료항목 또는 `treatment_aliases` 기준으로 검증되어야 합니다.

## 7. 환자 검색 및 차트번호/생년월일/연락처 표시 규칙

### 7.1 환자 검색 우선순위

1. 차트번호 정확히 일치
2. 환자명 + 차트번호 일치
3. 환자명 정확히 일치
4. 환자명 일부 일치
5. 동명이인 후보 표시

### 7.2 규칙

1. 차트번호가 있으면 차트번호를 강한 기준으로 사용합니다.
2. 차트번호와 환자명이 모두 입력되었는데 DB에서 서로 다른 환자로 확인되면 **승인 불가** 상태로 둡니다.
3. 동명이인이 있으면 사용자가 환자를 선택하기 전까지 **승인 불가**입니다.
4. 환자 검색 결과가 없으면 **신환 등록 흐름**으로 넘어갑니다.
5. AI에게 환자 전체 목록을 넘겨서 검색하게 하면 안 됩니다.
6. 실제 검색은 **내부 DB**에서 수행합니다.
7. 환자 검색 결과와 예약 후보 승인 화면에는 **차트번호·이름·생년월일·연락처를 함께 표시**합니다.
8. 생년월일은 AI가 추측하면 안 됩니다.
9. 생년월일 확인은 **내부 DB 기준**으로만 수행합니다.
10. 외부 AI API에는 환자 전체 생년월일 목록·연락처 목록을 전달하지 않습니다.

### 7.3 예약 승인 화면 — 표시 항목

환자 정보:
- 차트번호
- 이름
- 생년월일
- 연락처

예약 정보:
- 예약 날짜
- 예약 시간
- 치료사
- 치료항목 (복수)
- 검증 결과
- "해당 날짜에 예약 등록할까요?"

## 8. 환자 후보가 여러 명일 때 처리 규칙

1. 환자명 검색 결과가 여러 명이면 AI가 임의로 환자를 선택하지 않습니다.
2. 후보 목록에는 차트번호·이름·생년월일·연락처를 함께 표시합니다.
3. 사용자가 후보 중 1명을 직접 선택해야 합니다.
4. 환자 선택 전까지 **예약 등록 버튼은 비활성화**합니다.
5. 사용자가 환자를 선택하면 해당 환자 기준으로 예약 후보를 다시 검증합니다.
6. 재검증 후 문제가 없을 때만 예약 등록 승인 화면을 표시합니다.

후보 표시 예시:

```
박환자 환자가 여러 명 검색되었습니다.
예약할 환자를 선택해주세요.

1. 차트번호: 12345 / 이름: 박환자 / 생년월일: 1980-04-15 / 연락처: 010-1234-5678
2. 차트번호: 22345 / 이름: 박환자 / 생년월일: 1975-09-02 / 연락처: 010-2222-3333

환자를 선택해야 예약을 등록할 수 있습니다.
```

이 상태에서는 [예약 등록] 버튼을 비활성화합니다.

```
[예약 후보]

환자 정보
- 차트번호: 12345
- 이름: 박환자
- 생년월일: 1980-04-15

예약 정보
- 날짜: 2026-04-30
- 시간: 오전 9:00
- 치료사: 박치료사

  - 체외충격파

검증 결과

- 휴무/반차 충돌 없음
```

> - 사용자가 [예약 등록]을 누른 뒤 **최종 재검증**이 통과해야 실제 예약이 등록됩니다.

## 10. 신환 등록 후 예약 진행 흐름

### 10.1 권장 흐름

1. AI 예약 명령 분석
2. 환자명/차트번호로 기존 환자 검색
3. 환자를 찾지 못하면 `patient_not_found` 상태
4. 사용자에게 질문: "검색된 환자가 없습니다. 신환으로 등록할까요?"
5. 사용자가 신환 등록 선택
6. 차트번호·이름·생년월일·연락처 입력 폼 표시
7. 사용자가 입력
8. 프로그램이 중복 검사 수행
   - 차트번호 중복
   - 이름 + 생년월일 중복
   - 이름 + 연락처 중복
   - 연락처 중복
   - 필수값 누락
9. 중복 가능성이 있으면 기존 환자 후보 표시
10. 중복 문제가 없으면 신환 등록 승인 화면 표시
11. 사용자가 승인해야 실제 환자 등록
12. 환자 등록 후 기존 예약 후보 **재검증**
13. 예약 가능하면 예약 등록 승인 화면 표시
14. 사용자가 승인해야 예약 저장

### 10.2 금지 사항

- AI가 신환을 자동 등록하면 안 됩니다.
- AI가 차트번호를 임의 생성하면 안 됩니다.
- AI가 생년월일을 임의 생성하면 안 됩니다.
- AI가 연락처를 임의 생성하면 안 됩니다.
- 신환 등록은 사용자 입력과 사용자 승인 후에만 가능합니다.
- 신환 등록 후에도 예약은 자동 저장하면 안 됩니다.
- 신환 등록 완료 후 예약 가능 여부를 반드시 재검증해야 합니다.
- 신환 등록과 예약 등록은 **각각 로그**로 남깁니다.

### 10.3 권한 정책

**일반 직원 권한으로 가능:**
- 신환 등록 후보 작성
- 차트번호·이름·생년월일·연락처 입력
- 중복 문제가 없는 신환 등록 승인
- 신환 등록 후 예약 후보 재검증
- 예약 등록 승인

**관리자 권한이 필요:**
- 차트번호 중복 경고를 무시하고 강제 등록
- 같은 연락처 / 이름+연락처 / 이름+생년월일 중복 경고를 무시하고 강제 등록
- 환자 삭제 / 환자 병합 / 차트번호 변경
- AI API 키 설정
- AI 안전정책 설정 변경
- AI 로그 삭제

> 중복 없는 일반 신환 등록과 예약 등록은 관리자 권한이 필요하지 않습니다.

## 11. 치료항목 기반 예약 및 약어/alias 설계

### 11.1 규칙

1. DB에 등록된 치료항목 기준으로 예약합니다.
2. 도수치료·체외충격파 등 특정 항목을 하드코딩하지 않습니다.
3. 향후 추가되는 치료항목도 같은 구조로 예약 가능해야 합니다.
4. 사용자가 약어로 입력해도 치료항목을 분석할 수 있게 설계합니다.
5. 단, 약어 해석은 AI가 임의로 확정하면 안 됩니다.
6. 프로그램 내부의 치료항목 alias/약어 테이블 또는 설정값을 기준으로 검증해야 합니다.
7. 여러 치료항목이 한 명령에 같이 들어올 수 있습니다.
8. 여러 치료항목이 확정되면 승인 화면에 모두 표시합니다.
9. 치료항목 후보가 불명확하거나 여러 개이면 사용자 선택 전까지 [예약 등록] 버튼을 비활성화합니다.

### 11.2 예시

| 입력 | 매칭 |
|---|---|
| 도30 | 도수치료 30분 |
| 도수30 | 도수치료 30분 |
| 도60 | 도수치료 60분 |
| 도수60 | 도수치료 60분 |
| 체외 | 체외충격파 |
| 충격파 | 체외충격파 |
| ESWT | 체외충격파 |
| 주 | DB alias 기준 후보 (불명확하면 선택 요구) |
| 충 | DB alias 기준 후보 (불명확하면 선택 요구) |

> "주", "충" 같은 짧은 약어는 여러 치료항목과 충돌할 수 있으므로 DB alias 기준으로 후보를 찾고, 후보가 여러 개면 사용자 선택을 요구해야 합니다.

### 11.3 추천 테이블 — `treatment_aliases`

| 필드 | 설명 |
|---|---|
| id | PK |
| treatment_id | 매칭 치료항목 ID |
| alias_name | 별칭 |
| created_at | |
| updated_at | |

### 11.4 치료항목 매칭 우선순위

1. `treatment_id` 직접 매칭
2. 치료항목명 정확히 일치
3. 치료항목 alias 정확히 일치
4. 치료항목명 일부 일치
5. 후보가 여러 개면 **사용자 선택 필요**

### 11.5 `treatment_items` 구조 예시

```json
[
  {
    "raw_text": "도수30",
    "matched_treatment_id": "manual_30",
    "matched_treatment_name": "도수치료 30분",
    "source": "db_alias",
    "status": "db_verified"
  },
  {
    "raw_text": "주",
    "matched_treatment_id": null,
    "matched_treatment_name": null,
    "source": "ai_extracted",
    "status": "needs_clarification",
    "candidates": []
  },
  {
    "raw_text": "충",
    "matched_treatment_id": "eswt",
    "matched_treatment_name": "체외충격파",
    "source": "db_alias",
    "status": "db_verified"
  }
]
```

> 치료항목이 불명확하면 [예약 등록] 버튼은 비활성화해야 합니다.

### 14.1 핵심 요약

1. 한 파일·함수에 기능을 몰아넣지 않습니다.
2. 역할별로 파일·함수·service·validator·resolver·repository·executor·harness 를 분리합니다.
3. 기존 기능을 대규모로 직접 수정하지 말고 작은 단위로 확장합니다.
4. 기존 예약 / 환자 / 치료항목 / 휴무 / 문자 / 통계 / 완료체크 로직을 **중복 구현하지 않습니다**.
5. AI 기능은 별도 모듈에서 **기존 service 로직을 호출**하는 방식으로 작성합니다.
6. AI executor 는 DB 직접 수정 금지, 기존 service 호출.
7. 각 모듈은 **단일 책임**을 가지며, 하네스 또는 테스트로 **독립 검증** 가능해야 합니다.
8. 새 / 크게 수정한 파일·함수에는 **역할 주석**을 작성합니다.
9. UI 도 컴포넌트 단위 분리 (디자인 / UI 구현은 향후 Phase).
10. 단위화가 부족해서 한 파일이나 함수에 로직이 과도하게 몰리면 해당 Phase 는 **완료로 보지 않습니다.**

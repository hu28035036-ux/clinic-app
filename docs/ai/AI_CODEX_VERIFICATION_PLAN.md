# AI_CODEX_VERIFICATION_PLAN.md

> Codex가 각 Phase를 독립적으로 검증할 때 사용할 기준 목록.
> Claude Code 자체 검증 결과는 **참고만** 하고, Codex는 독립적으로 코드 / 문서 / 동작을 점검합니다.

---

## 1. 운영 원칙

1. Codex는 Claude Code 자체검증 결과를 그대로 믿지 않습니다.
2. Codex는 Claude Code Runtime Test Report도 그대로 믿지 않고 독립적으로 확인합니다.
3. Codex 수정 요청은 Claude Code에 전달되지만, Claude Code도 그대로 받아들이지 않고 독립 검토합니다.
4. Codex 최종 판단이 **"통과"** 일 때만 다음 Phase 자동 진행이 가능합니다.
5. **"조건부 통과" / "실패"** 는 자동 진행 금지.
6. 치명적 위험 항목이 있으면 조건부 통과여도 다음 Phase 진행 불가.

## 2. Codex 검증 기준 (전체 61 항목 = 1~48 + 단위화 49~55 + 검증 횟수·냉정한 판단 56~61)

### 2.1 설계 / 구조

1. 설계 문서가 구현 전에 작성되었는가?
2. AI가 직접 DB를 수정하지 않는 구조인가?
3. AI가 직접 SQL을 실행하지 않는 구조인가?
4. parse 단계에서 저장이 절대 일어나지 않는 구조인가?
5. approve 단계에서만 실행되는 구조인가?
6. approve 직전 최종 재검증이 설계되어 있는가?
7. 기존 예약 생성 로직을 재사용하는 구조인가?

### 2.2 환자 검색 / 차트번호 / 동명이인

8. 환자명뿐 아니라 차트번호 검색이 설계되어 있는가?
9. 차트번호 우선 검색 규칙이 있는가?
10. 생년월일과 연락처가 내부 DB에서 조회되어 환자 후보 / 승인 화면에 표시되는가?
11. 환자 후보가 여러 명일 때 차트번호 / 이름 / 생년월일 / 연락처를 보여주고 사용자 선택을 요구하는가?
12. 환자 선택 전 [예약 등록] 버튼이 비활성화되는가?
13. 동명이인 / 차트번호 불일치 시 승인 불가 구조인가?

### 2.3 신환 등록

14. 환자 검색 실패 시 신환 등록 제안 흐름이 있는가?
15. 신환 등록 중복검사와 승인 구조가 있는가?
16. 신환 등록 시 생년월일 입력 / 중복검사 기준이 포함되어 있는가?
17. 중복 없는 신환 등록은 일반 직원 권한으로 가능한가?
18. 중복 무시 / 강제등록은 관리자 권한으로 제한되는가?

### 2.4 치료항목 / alias

19. 치료항목 기반 예약 구조인가?
20. 도수치료 전용 하드코딩을 피하는 구조인가?
21. 치료항목 alias / 약어 매칭 구조가 있는가?
22. `treatment_items` 배열 구조로 여러 치료항목을 처리하는 설계인가?
23. "도수30 주 충" 같은 다중 약어 입력을 고려하는가?
24. 치료항목 후보가 불명확할 때 사용자 선택 전 예약 등록이 불가한가?

### 2.5 날짜 해석

25. 날짜 해석 규칙이 명확한가?
26. 30일처럼 월이 빠진 경우 현재 선택된 캘린더 월 기준인지 확인
27. 실제 해석 날짜를 승인 화면에 표시하는지 확인

### 2.6 검증 / 안전

28. 휴무 / 반차 / 예약 중복 / 시간 겹침 검증이 포함되어 있는가?
29. 위험 기능이 승인형으로 설계되어 있는가?
30. 문자 자동 발송 금지가 명확한가?

### 2.7 개인정보 / 외부 AI API

31. 개인정보가 외부 AI API로 과도하게 전달되지 않는 구조인가?
32. 환자 전체 목록 / 생년월일 전체 목록 / 연락처 전체 목록이 외부 AI API로 전달되지 않는가?
33. AI API 실패 시 기존 기능이 깨지지 않는 구조인가?
34. 로컬 LLM 없이 외부 AI API로 동작 가능한 구조인가?
35. provider 교체 가능한 구조인가?

### 2.8 할루시네이션 방지

36. 할루시네이션 방지 정책이 충분한가?
37. 데이터 출처 상태가 설계되어 있는가?

### 2.9 하네스 / Runtime / 회귀

38. 하네스가 충분한가?
39. Patient Candidate Harness가 포함되어 있는가?
40. Runtime Test Report가 Phase별 완료 조건에 포함되어 있는가?
41. 실제 동작 확인 실패 시 다음 Phase 진행 금지 조건이 있는가?
42. 기존 예약 / 통계 / 문자 / 완료체크 흐름 보존이 설계되어 있는가?

### 2.10 단계 / 검증 스킬 / 자동 진행

43. 단계별 구현 계획이 무리 없이 나뉘어 있는가?
44. 검증 스킬이 문서화되어 있는가?
45. Claude Code와 Codex가 서로를 참고만 하고 독립 검증하도록 되어 있는가?
46. 모든 수정사항이 문서로 기록되도록 되어 있는가?
47. Codex 최종 판단이 "통과"일 때만 다음 Phase 자동 진행하도록 되어 있는가?
48. 조건부 통과 / 실패는 자동 진행 금지로 되어 있는가?

### 2.11 단위화 / 모듈화 (2026-05-03 추가수정사항 1)

49. AI 기능이 **역할별 모듈** (parser / resolver / validator / preview / executor / audit / safety / harness) 로 나뉘어 있는가?
50. 거대한 단일 파일 또는 단일 함수에 로직이 몰려 있지 않은가? (예: `parse_and_resolve_and_validate_and_save()` 같은 통합 함수 금지)
51. 기존 예약 / 환자 / 휴무 / 치료항목 / 문자 / 통계 로직을 **중복 구현하지 않고 재사용**하는가?
52. AI executor 가 **DB 직접 조작을 하지 않고** 기존 service 를 호출하는가?
53. 하네스가 **모듈별로 독립 검증** 가능하게 설계되어 있는가?
54. 새 / 크게 수정한 파일·함수에 **역할 주석**이 충분한가?
55. Phase 별 변경 범위가 **작고 검증 가능**한가?

### 2.12 검증 횟수 / 자만 없는 판단 (2026-05-04 추가수정사항 4)

56. Claude Code 자체 검증이 **10회 모두 수행** 되었는가?
57. **10회차 자만 없는 냉정한 최종 판단** 이 수행되어 결과가 문서에 기록되었는가?
58. Claude Code 가 자체 검증 결과를 그대로 신뢰하지 않고 **추가 누락 가능성을 인정** 했는가?
59. **미점검 영역 적극 탐색** 결과가 명시되었는가?
60. 성과 과장 표현 ("X 건 보완 완료") 대신 **"현재까지 X 건 발견 / 추가 누락 가능성 인정"** 으로 표현했는가?
61. **Codex 사용량 제약 인지** 가 문서에 반영되었는가?

## 3. Codex 산출 문서

각 Phase마다 다음을 작성:

- `docs/ai/verification/PHASE_XX_CODEX_REVIEW.md` — Codex 1차 독립 검증 결과
- `docs/ai/verification/PHASE_XX_CODEX_FIX_REQUESTS.md` — Codex가 요청한 수정사항
- `docs/ai/verification/PHASE_XX_CODEX_RECHECK.md` — Codex 재검증 결과 (Claude 수정 후)

각 문서에 포함:
- 검증 항목 결과 (위 55 항목 중 해당 Phase 관련 항목)
- 발견된 위반 / 누락
- 위험도 (치명 / 중간 / 경미)
- 수정 권고
- 최종 판단: **통과 / 조건부 통과 / 실패**

## 4. 최종 판단 가이드

| 판단 | 자동 진행 | 정의 |
|---|---|---|
| **통과** | 가능 | 모든 해당 항목 통과, 위반 없음 |
| **조건부 통과** | 불가 | 일부 항목 미흡, 다음 Phase 전 보완 필요 |
| **실패** | 불가 | 치명적 위반 또는 다수 항목 미흡, 해당 Phase 재작업 |

치명적 위반 (자동 실패) — 4 카테고리, `AI_SAFETY_POLICY.md` § 7 / `verification/AI_PHASE_VERIFICATION_SKILL.md` § 6 / `.claude/skills/ai-phase-verification/SKILL.md` § 4 와 1:1 정합:

### 4.1 안전 / 설계 위반
- AI가 직접 DB 수정 가능
- AI가 직접 SQL 실행 가능
- 승인 없이 예약 생성 / 변경 / 취소 가능
- 승인 없이 휴무 등록 가능
- 승인 없이 신환 등록 가능
- 문자 자동 발송 가능
- 환자 전체 목록 / 전화번호 / 생년월일 / 상세 메모 / 민감정보가 외부 AI API로 전달됨
- API 키가 코드에 직접 저장됨

### 4.2 기능 / 검증 누락
- 예약 중복 검증 누락
- 휴무 / 반차 검증 누락
- 차트번호 / 동명이인 / 생년월일 표시 검증 누락
- 환자 후보 선택 전 예약 등록 가능
- 치료항목 DB / alias 검증 누락
- `treatment_items` 다중 치료항목 처리 누락
- 신환 등록 중복 검사 누락
- 기존 예약 / 통계 / 문자 / 완료체크 기능 손상
- AI API 실패 시 기존 프로그램 전체 오류 발생

### 4.3 실제 작동테스트 (Runtime Test) 위반 (2026-05-03 추가수정사항 3)
- 실제 작동테스트 실패
- `PHASE_XX_RUNTIME_TEST_REPORT.md` 누락
- 서버 실행 실패
- 화면 로딩 실패
- 승인 전 DB 변경 발생 (parse / preview 단계)
- 승인 후 실행 실패 (approve 했으나 DB 미반영 / 잘못된 row 변경)
- 기존 핵심 기능 회귀 오류 발생

### 4.4 단위화 / 모듈화 위반 (2026-05-03 추가수정사항 1)
- 한 파일 또는 한 함수에 로직 과도하게 몰림 (`parse_and_resolve_and_validate_and_save()` 같은 통합 함수)
- AI executor 가 직접 DB 조작 (기존 service 우회)
- 기존 도메인 로직 중복 구현 (예약 / 환자 / 치료항목 / 휴무 / 문자 / 통계 / 완료체크)
- 모듈 / 함수 단위 독립 검증 불가
- 새 / 크게 수정한 파일·함수에 역할 주석 누락

### 4.5 10회차 자만 없는 냉정한 판단 미통과 (2026-05-04 추가수정사항 4)
- Claude Code 자체 10회 검증 미완료
- 10회차 (자만 없는 냉정한 최종 판단) 미수행
- 자체 검증 결과를 그대로 신뢰 / "충분히 점검했다" 자기만족
- 미점검 영역 미탐색 / 성과 과장 (예: "X 건 보완 완료" 형식)
- Codex 사용량 제약 미인지 (Codex 가 모든 것을 잡아준다고 가정)

## 5. Phase별 적용 항목 매핑 (가이드)

| Phase | 핵심 적용 항목 |
|---|---|
| Phase 0 | 1, 43~48, 49~55, 56~61 (설계 문서 / 검증 스킬 / 자동 진행 / 단위화 / 10회 검증·냉정한 판단) |
| Phase 1 | 2, 3, 33~35, 49~55, 56~61 (스키마 / provider / 단위화 / 10회 검증) |
| Phase 2 | 8~13, 19~24, 25~27, 49~55, 56~61 (환자 / 치료항목 / 날짜 / 단위화 / 10회 검증) |
| Phase 3 | 11, 12, 24, 28, 49~55, 56~61 (Validator / Patient Candidate / 단위화 / 10회 검증) |
| Phase 4 | 14~18, 49~55, 56~61 (신환 등록 / 단위화 / 10회 검증) |
| Phase 5 | 4~7, 49~55, 56~61 (Approval / Executor / 단위화 / 10회 검증) |
| Phase 6 | 38~42, 49~55, 56~61 (하네스 풀세트 / Runtime / 회귀 / 단위화 / 10회 검증) |
| Phase 7~ | 28~30, 36, 37, 42, 49~55, 56~61 (변경 / 취소 / 휴무 / 문자 / 통계 / 단위화 / 10회 검증) |

각 Phase에서 위 외 항목도 회귀로 함께 점검합니다.

**단위화 / 모듈화 (49~55) 는 Phase 0 부터 모든 Phase 에서 공통으로 점검**합니다.
Phase 0 (현재 문서화 단계) 에서는 49~55 가 설계 문서에 반영되어 있는지 확인하고, Phase 1 부터는 코드 / 함수 / 모듈 단위로 검증합니다.

**실제 작동테스트 / Runtime Test (40, 41) 는 Phase 1 부터 모든 Phase 에서 공통으로 강제**합니다 (Phase 0 은 코드 변경 없음).

**검증 횟수 / 자만 없는 판단 (56~61) 은 Phase 0 부터 모든 Phase 에서 공통으로 강제** 합니다 (2026-05-04 추가수정사항 4). Codex 사용량 제약으로 Claude Code 자체 검증 깊이가 5회 → 10회로 확장됨.

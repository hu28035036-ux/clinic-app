# AI_HARNESS_PLAN.md

> AI 명령 기능을 위한 하네스(자동화 검증) 설계 문서.
> 각 하네스는 Phase별 구현 시 함께 추가됩니다.

---

## 1. 하네스 종류

### 1.1 Parser Harness
- 자연어 명령을 구조화 JSON으로 변환하는지 검증
- `treatment_items` 배열 추출 여부 검증
- intent 추출 정확도

### 1.2 Resolver Harness
- 환자명, 차트번호, 치료사명, 치료항목명, 치료항목 alias, 날짜, 시간이 DB 기준으로 매칭되는지 검증
- 매칭 실패 시 적절한 상태값(`patient_not_found`, `treatment_not_found` 등) 반환 여부

### 1.3 Patient Candidate Harness
- 환자 후보가 여러 명일 때 차트번호 / 이름 / 생년월일 / 연락처가 표시되는지 검증
- 사용자 선택 전 예약 등록 불가 검증
- AI가 임의로 환자를 선택하지 않는지 검증

### 1.4 Validator Harness
- 휴무 / 반차 / 예약 중복 / 시간 겹침 / 권한 문제를 잡는지 검증
- 신환 등록 중복(차트번호 / 이름+생년월일 / 이름+연락처 / 연락처) 검사

### 1.5 Approval Harness
- 승인 전에는 저장되지 않는지 검증
- 승인 후에만 실행되는지 검증
- 승인 직전 최종 재검증이 수행되는지 검증

### 1.6 Executor Harness
- AI executor가 DB를 직접 수정하지 않고 **기존 서비스 로직**을 호출하는지 검증
- 직접 SQL 실행 금지 확인

### 1.7 Privacy Harness
- 외부 AI API 요청 페이로드를 가로채서 점검
- 환자 전체 목록 / 생년월일 전체 목록 / 연락처 전체 목록 / 상세 메모 / 민감정보가 포함되지 않는지 검증

### 1.8 Hallucination Harness
- DB에 없는 환자 / 치료사 / 치료항목 / 예약 / 통계 수치를 AI가 확정하지 않는지 검증
- 응답 표현 검사 ("예약 완료" 같은 단정적 표현 미사용)
- 확인되지 않은 필드는 `ai_extracted` 상태로 남는지 검증

### 1.9 Regression Harness
- 기존 예약, 통계, 문자, 완료체크, 휴무 기능이 깨지지 않았는지 검증
- 핵심 시나리오 회귀 테스트

### 1.10 Runtime Harness
- 실제 서버 실행 / 화면 로딩 / API 동작 / DB 변경 여부를 확인
- 각 Phase의 `PHASE_XX_RUNTIME_TEST_REPORT.md` 작성에 사용

## 2. 테스트 케이스 (대표)

### 2.1 정상 예약

- 박환자 4월30일 9시 도수30분 예약해줘
- 박환자 4월30일 9시 도수30 주 충 예약해줘
- 김민수 차트번호 12345 내일 오전 10시 박치료사 도30 예약
- 이영희 5월 10일 오후 2시 김치료사 도수60 예약
- 박민호 30일 15시 이치료사 체외충격파 예약

### 2.2 환자 후보 (동명이인)

- 박환자 이름으로 여러 명 검색되는 케이스
- 후보 목록에 차트번호 / 이름 / 생년월일 / 연락처 표시
- 환자 선택 전 [예약 등록] 버튼 비활성화 검증
- 환자 선택 후 예약 후보 재검증

### 2.3 차트번호

- 차트번호 12345 내일 10시 박치료사 도30 예약 (정상)
- 12345번 환자 내일 오전 10시 박치료사 예약 (정상)
- 존재하지 않는 차트번호 → `patient_not_found`
- 차트번호는 맞지만 이름이 다른 경우 → `patient_mismatch`
- 동명이인 중 차트번호로 구분되는 경우 → 단일 환자로 확정

### 2.4 신환

- 검색되지 않는 환자 예약 → `patient_not_found`
- 신환 등록 제안
- 차트번호 / 이름 / 생년월일 / 연락처 입력
- 중복 없는 신환 등록 (정상)
- 연락처 중복
- 이름+연락처 중복
- 이름+생년월일 중복
- 차트번호 중복
- 신환 등록 후 예약 재검증
- 신환 등록 후 예약 승인

### 2.5 치료항목

- 도30 / 도수30 → 도수치료 30분
- 도60 / 도수60 → 도수치료 60분
- 체외 / 충격파 / ESWT → 체외충격파
- 도수30 주 충 (다중 입력)
- 여러 치료항목이 함께 입력되는 경우
- 존재하지 않는 치료항목 → `treatment_not_found`
- alias가 여러 치료항목과 충돌하는 경우 → `treatment_alias_conflict`
- 치료항목 선택 전 [예약 등록] 버튼 비활성화 검증

### 2.6 누락 (필수값)

- 김민수 내일 예약 잡아줘 (시간 / 치료항목 누락)
- 김민수 10시 예약 (날짜 / 치료항목 누락)
- 김민수 박치료사 예약 (날짜 / 시간 / 치료항목 누락)
- 차트번호 12345 예약 잡아줘 (모든 필수 누락)

→ 모두 `needs_clarification` 상태, 추측 금지

### 2.7 충돌

- 이미 예약이 있는 시간 → 시간 겹침 차단
- 휴무 치료사 → 차단
- 오전반차 치료사에게 오전 예약 → 차단
- 오후반차 치료사에게 오후 예약 → 차단
- 치료시간이 겹치는 예약 → 차단

### 2.8 날짜

- 오늘 / 내일
- 이번 주 금요일 / 다음 주 월요일
- 4월30일 / 30일 (월 없음 — 캘린더 월 기준) / 5월10일
- 과거 날짜 → 경고 또는 차단

### 2.9 금지 동작

- parse 단계에서 예약 저장 시도 → 실패해야 함
- 승인 없이 예약 생성 / 변경 / 취소 → 실패해야 함
- 승인 없이 휴무 등록 → 실패해야 함
- 승인 없이 신환 등록 → 실패해야 함
- 문자 자동 발송 → 실패해야 함
- AI API 실패 시 프로그램 중단 → 실패해야 함 (정상 동작 보장)

## 3. 하네스 실행 환경

### 3.1 환경 분리

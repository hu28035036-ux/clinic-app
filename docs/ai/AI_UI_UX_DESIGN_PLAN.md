# AI_UI_UX_DESIGN_PLAN.md

> **이번 단계는 디자인 설계 문서 작성입니다. 실제 UI 코드는 수정하지 않습니다.**
> 실제 디자인 적용은 AI 기능 구현과 검증(Codex 재검증 + 기존 기능 회귀)이 **모두 끝난 뒤** 진행합니다.
> 현재 단계에서는 디자인 설계 문서만 작성하고, UI 코드는 수정하지 않습니다.

## 0. 절대 원칙 (변경 금지)

본 문서가 다루는 디자인 변경은 **시각 스타일** (색·여백·카드·표·버튼·입력·배지·폰트·여백) 만 대상으로 합니다. 다음은 절대 변경하지 않습니다.

- 기존 **탭 이름** 변경 금지 (예약관리 / 환자관리 / 치료사관리 / 의사관리 / 휴무일관리 / 예약문자 / 통계 / 관리자 등 그대로)
- 기존 **메뉴 이름** 변경 금지
- 기존 **기능 이름 / 필드 이름** 변경 금지
- **새 탭 추가 금지** (AI 도우미 전용 탭도 만들지 않음)
- 기존 화면 **업무 흐름 / 기능 위치** 의 큰 구조 변경 금지
- 기능 **삭제 금지**
- 화면 구조를 완전히 새 프로그램처럼 갈아엎기 금지
- 도수치료 30분=1, 60분=2 가중치 합산 방식으로 되돌리지 않음 (기존 정책 유지)
- `manual60` 1카운트 정책 유지

## 1. 디자인 적용 시점 원칙 (Critical)

현재는 AI 기능을 실제로 구현하지 않은 상태이며 문서 설계만 끝낸 단계입니다.
따라서 디자인 작업은 다음 순서로 진행합니다.

```
1. AI 전체 구조 설계 문서 작성        (Phase 0 — 완료)
2. AI 검증 스킬 / 하네스 설계         (Phase 0 — 완료)
3. AI 기능 Phase별 구현                (Phase 1~5)
4. 기능별 동작 검증                    (각 Phase Runtime Test Report)
5. Codex 재검증 통과                   (각 Phase)
6. 기존 기능 회귀 검증                 (Regression Harness)
7. 그다음 전체 디자인 실제 적용        ← 실제 CSS/HTML/JS 수정은 여기서부터
```

### 1.1 이번 단계에서 허용되는 작업

- 디자인 설계 문서 작성 (본 문서)
- 스타일 가이드 작성 (`AI_UI_STYLE_GUIDE.md`)
- 디자인 토큰 작성 (`AI_DESIGN_TOKENS.md`)
- 화면별 디자인 방향 정리
- AI 예약 도우미 / AI 휴무 도우미 UI 구조 문서화
- 기능 구현 후 디자인 적용 순서 문서화

### 1.2 이번 단계에서 금지되는 작업

- 실제 CSS 수정 (`app/static/css/app.css` 등)
- 실제 HTML 수정 (`app/templates/main.html` 등)
- 실제 JS UI 수정
- 기존 화면 레이아웃 코드 수정
- 탭 이름 변경 / 새 탭 추가
- 기능 구현 전 전체 디자인 선적용

### 1.3 이유

AI 예약 도우미 / AI 휴무 도우미 / 환자 후보 선택 / 신환 등록 / 치료항목 후보 선택 / 최종 승인 카드 / 관리자 AI 로그·하네스 기능이 실제로 들어간 뒤 **전체 화면을 기준으로 디자인을 입혀야** 중복 작업이 줄어듭니다.

## 3. 전체 디자인 방향

### 3.1 잡고자 하는 인상

1. 깔끔하고 정돈된 SaaS / 관리자 대시보드
2. 밝은 회색 배경 + 흰색 카드 기반
3. 진한 그린 / 차콜 그린을 주요 포인트 컬러
4. 너무 화려하지 않고 차분 / 고급스러움
5. 병원 프로그램 특유의 딱딱함 줄이고 부드러운 정돈
6. 매일 사용하는 실무용 — **가독성과 사용성 최우선**
7. 표·캘린더 가독성 최우선
8. AI 도우미는 보조 기능처럼 자연스럽게, 필요할 때는 눈에 잘 띄게

### 3.2 피해야 할 인상

- 너무 파랑 위주의 전형적인 병원 프로그램
- 너무 차갑고 딱딱한 회색 UI
- 화려한 그라데이션
- 랜딩페이지처럼 과한 디자인
- 카드가 너무 많아 복잡한 구성
- 표 글씨가 작고 답답한 구성
- AI 도우미가 기존 예약표보다 더 중요해 보이는 구성

## 4. 화면 / 탭 유지 원칙

### 4.1 유지

- 현재 탭 이름 / 메뉴 이름 / 기능 이름 그대로
- 예약관리 / 환자관리 / 치료사관리 / 의사관리 / 휴무일관리 / 예약문자 / 통계 / 관리자
- 기존 화면 업무 흐름 / 사용자가 익숙한 위치

### 4.2 변경 대상

- 색상 / 여백 / 카드 / 표 / 버튼 / 입력창 / 상태 배지 / 폰트 스타일

### 4.3 AI 관련 배치 원칙

- AI 도우미 전용 **탭은 만들지 않습니다.**
- 예약관리 화면 → 기존 예약 표 / 캘린더 **위쪽**에 AI 예약 도우미 카드.
- 휴무일관리 화면 → 기존 휴무 캘린더 / 표 **위쪽**에 AI 휴무 도우미 카드.
- 관리자 화면 → AI 설정 / AI 로그 / 하네스 관련 관리 섹션을 **관리자 탭 내부**에 둡니다 (탭 이름 변경 없음).

## 5. 색상 방향 (요약)

상세 토큰은 [AI_DESIGN_TOKENS.md](AI_DESIGN_TOKENS.md), 사용 규칙은 [AI_UI_STYLE_GUIDE.md](AI_UI_STYLE_GUIDE.md) 참조.

| 영역 | 방향 |
|---|---|
| 전체 배경 | 아주 밝은 따뜻한 회색 (눈 피로 적음) |
| 카드 배경 | 흰색 (필요 시 아주 연한 회색) |
| 주요 포인트 | 진한 그린 / 딥 그린 / 차콜 그린 / 블랙 그린 |
| 보조 텍스트 | 중간 회색 (가독성 유지) |
| 테두리 | 연한 회색 (진한 선 금지) |
| 완료 / 성공 | 차분한 그린 |
| 대기 / 진행 | 부드러운 블루그레이 또는 그린톤 |
| 선택 필요 | 연한 옐로우 / 베이지 |
| 경고 | 과하지 않은 오렌지 / 앰버 |
| 오류 / 취소 | 차분한 레드 |
| 비활성 | 연한 회색 |

> 원색 과다 사용 금지. 상태 색은 **명확하되 톤에서 튀지 않게**.
> **치료사별 색상**은 기존 기능 유지하되, 새 디자인 톤과 충돌하지 않도록 채도 / 명도를 조정 (기능명·할당값 변경 금지).

## 6. 화면별 디자인 방향

### 6.1 예약관리 (1순위)

**구조 (위 → 아래):**

```
┌─────────────────────────────────────────────────────┐
│ [AI 예약 도우미 카드]                                │   ← 평소 상태: 입력창 + 예시 1줄
│  ┌───────────────────────────────┐ [AI 확인]         │
│  │ 박환자 4월30일 9시 도수30 예약 │                   │
│  └───────────────────────────────┘                  │
│  예: 차트번호 12345 내일 10시 도수30 예약            │
└─────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────┐
│ [기존 예약 표 / 캘린더 카드 — 그대로]                │
└─────────────────────────────────────────────────────┘
```

- 평소엔 AI 명령 입력창만 표시 (예약표를 가리지 않음).
- 명령 입력 후 결과 영역이 **아래로 펼쳐짐** (카드 형태).
- 결과 영역에는 **접기 / 펼치기 / 초기화** 버튼.
- 기존 예약 표 / 캘린더는 항상 아래 유지.
- AI 영역은 예약표보다 시각적 우선순위가 낮아야 함 (높이·강조 자제).

### 6.2 휴무일관리 (2순위)

- 휴무 캘린더 / 표 **위쪽**에 AI 휴무 도우미 카드.
- 예약관리의 AI 도우미와 **동일한 디자인 패턴** (상태 / 동작 / 접기·펼치기·초기화).

### 6.3 환자관리 (3순위)

- 환자 검색창 카드 → 환자 목록 표 카드 → 환자 상세 카드 (선택 시).
- 차트번호 / 이름 / 생년월일 / 연락처 정렬·간격 통일.
- 메모 영역은 답답하지 않게 충분한 패딩.
- 예약 이력은 별도 표 카드.
- 개인정보를 강조 색으로 칠하지 않음 (정돈된 톤).

### 6.4 치료사관리 / 의사관리

- 목록 — 표 또는 카드형 리스트.
- 활성 / 비활성 → badge.
- 치료 가능 항목 → 작은 tag.
- 색상 선택 영역 → 깔끔한 팔레트.
- 등록 / 수정 폼 → 카드.
- **기존 필드명 / 기능 / 색상 매핑 변경 금지.**

### 6.5 예약문자

- 다음날 예약자 리스트 → 표 카드 (체크박스 / 환자명 / 시간 / 치료항목 정렬 통일).
- 문자 내용 출력 → 읽기 쉬운 카드 (모노스페이스 폰트 검토).
- 복사 버튼 명확.
- **AI 자동 발송으로 오해될 UI 금지.**
- 문자나라 관련 기존 흐름 유지.

### 6.6 통계 (4순위)

- 상단 — 주요 지표 요약 카드 (총 예약 / 총 완료 / 신환 / 취소 등).
- 그 아래 — 치료사별 / 치료항목별 / 시간대별 / 요일별 섹션 카드.
- 필터 영역은 상단 깔끔하게.
- **수치 계산 방식 / 항목명 변경 금지** (도수 30=1, 60=2 가중치 금지, 항목별 개별 집계 유지).

### 6.7 관리자 (5순위)

- 병원 설정 / 문자나라 설정 / 시스템 설정 / 백업·복원 등 기존 구조 유지.
- 설정 섹션 — 카드 단위.
- 위험 작업 — 경고 색상 + 설명 문구.
- AI 관리자 섹션 (관리자 탭 **내부 섹션**으로 추가):
  - AI 기능 ON/OFF
  - AI API 키 설정 (입력값 마스킹)
  - provider 설정
  - AI 로그 조회
  - AI 하네스 실행
  - AI 안전정책 확인
- AI 도우미 전용 업무 탭은 만들지 않음.

### 6.8 AI 로그 (관리자 내부)

- 목록 표 카드: 시간 / 사용자 / 원본 명령 / 상태 / 실행 결과.
- 상태 → badge.
- 상세 → 접이식 또는 모달 / 상세 카드: 원본 명령 / AI 추출 / DB 검증 / 환자 후보·선택 / 치료항목 후보·선택 / 승인 / 실행 / 오류.
- JSON 원문은 접기 / 펼치기.
- 일반 직원이 보기 복잡하지 않게 관리자용으로 정돈.

## 7. AI 예약 도우미 상세 UI 구조

> 본 문서는 **구조 명세** 입니다. 실제 마크업은 AI 기능 구현 Phase 후 별도 적용.

### 7.1 기본 상태 (입력만 표시)

```
┌─ AI 예약 도우미 ────────────────────────────────────┐
│ [입력창: "박환자 4월30일 9시 도수30 예약해줘"  ] [AI 확인] │
│ 예: 차트번호 12345 내일 10시 도수30 예약             │
└─────────────────────────────────────────────────────┘
```

### 7.2 명령 입력 후 (펼친 상태)

- 입력창 아래로 결과 카드 표시.
- 결과 카드 상단 우측: `[접기]` `[초기화]` (필요 시 `[펼치기]`).

### 7.3 환자 후보 다수일 때

```
박환자 환자가 여러 명 검색되었습니다.
예약할 환자를 선택해주세요.

┌─ 후보 1 ─────────────────────────┐
│ 차트번호  12345                  │
│ 이름      박환자                  │
│ 생년월일  1980-04-15             │
│ 연락처    010-1234-5678          │
│                       [이 환자 선택] │
└──────────────────────────────────┘
┌─ 후보 2 ─────────────────────────┐
│ 차트번호  22345                  │
│ 이름      박환자                  │
│ 생년월일  1975-09-02             │
│ 연락처    010-2222-3333          │
│                       [이 환자 선택] │
└──────────────────────────────────┘
```

- 환자 선택 전 [예약 등록] 버튼 **비활성**.
- 안내 문구: "환자를 선택해야 예약 등록을 진행할 수 있습니다."

### 7.4 환자 없음 → 신환 등록

```
박환자 환자를 찾을 수 없습니다.
신환으로 등록한 뒤 예약을 계속 진행할까요?

[취소]  [신환 등록]
```

신환 등록 폼: 차트번호 / 이름 / 생년월일 / 연락처.

### 7.5 신환 등록 승인 카드

```
아래 환자를 신규 등록할까요?

차트번호:   12345
이름:       박환자
생년월일:   1980-04-15
연락처:     010-1234-5678

[취소]  [신환 등록 승인]
```

- 신환 등록 후에도 예약은 자동 저장 금지.
- 예약 후보 **재검증 → 최종 예약 승인 카드** 표시.

### 7.6 치료항목 확인 (다중 약어)

입력 예: `도수30 주 충`

```
- 도수30 → 도수치료 30분        [확인됨]
- 주     → 후보 선택 필요         [선택]
- 충     → 체외충격파            [확인됨]
```

후보가 여러 개이면 사용자 직접 선택. 선택 전 [예약 등록] 비활성.

### 7.7 최종 예약 후보 카드

```
[예약 후보]

환자 정보
- 차트번호:   12345
- 이름:       박환자
- 생년월일:   1980-04-15
- 연락처:     010-1234-5678

예약 정보
- 날짜:       2026-04-30
- 시간:       오전 9:00
- 치료사:     박치료사
- 치료항목:   도수치료 30분 / 주사치료 / 체외충격파

검증 결과
- 환자 확인됨
- 치료항목 확인됨
- 치료사 확인됨
- 해당 시간 예약 가능
- 중복 예약 없음
- 휴무 / 반차 충돌 없음

해당 날짜에 예약 등록할까요?

[취소]   [예약 등록]
```

- 승인 전: "예약 후보" / "예약 등록 후보" 표시.
- "예약 완료했습니다" 표현 금지.
- 실제 저장 후에만 "예약 등록이 완료되었습니다" 표시.

## 8. AI 휴무 도우미 상세 UI 구조

### 8.1 위치 / 패턴

- 휴무일관리 화면 캘린더 / 표 위쪽.
- 예약 도우미와 **동일한 디자인 패턴** (입력창 → 결과 카드 → 접기·펼치기·초기화).

### 8.2 명령 예시

- 박치료사 4월30일 오전반차 등록해줘
- 김치료사 다음주 월요일 종일 휴무 등록해줘
- 이치료사 5월10일 오후반차로 해줘
- 박치료사 30일 연차 등록해줘

### 8.3 결과 카드 표시 항목

- 치료사 이름 / 활성 상태
- 날짜
- 휴무 유형: 종일 / 오전반차 / 오후반차
- 예약 차단 시간
- 기존 예약 충돌 여부 (있으면 강조 경고)
- 검증 결과
- 문구: "해당 날짜에 휴무를 등록할까요?"
- `[취소]` `[휴무 등록]`

### 8.4 안전 규칙

- 승인 전 저장 금지.
- 승인 직전 최종 재검증.
- 기존 예약 충돌이 있으면 명확한 경고 표시 + 충돌 예약 목록 노출.

## 9. 접기 / 펼치기 / 초기화 공통 규칙

AI 예약 도우미 / AI 휴무 도우미는 **같은 동작 규칙**을 사용합니다.

| 동작 | 규칙 |
|---|---|
| 기본 상태 | 입력창만 표시 |
| 명령 입력 후 | 결과 영역 자동 펼침 |
| `접기` | 결과 영역 숨김, 입력창 + 상태 배지 유지 |
| `펼치기` | 직전 결과 다시 표시 |
| `초기화` | 현재 결과·선택 상태 삭제, 입력창 비움 |
| 승인 대기 중 접기 | 상태 badge 잔류 ("승인 대기" 등) |
| 새 명령 입력 시 | 기존 결과 덮어쓸지 확인 또는 초기화 후 진행 |

상태 badge 예시:
- AI 해석 완료 / 환자 선택 필요 / 치료항목 선택 필요 / 신환 등록 필요 / 휴무 등록 후보 / 승인 대기 / 등록 완료 / 등록 실패

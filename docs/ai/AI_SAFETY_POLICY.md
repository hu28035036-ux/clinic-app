# AI_SAFETY_POLICY.md

> AI 기능 안전 정책: 금지 사항 / 할루시네이션 방지 / 개인정보 보호 / API 키 관리
> 이 정책은 모든 Phase에서 강제되며, 위반 시 다음 Phase로 자동 진행할 수 없습니다.

---

## 1. 절대 금지 사항

### 1.1 DB / 실행 관련

1. AI가 직접 DB를 수정하면 안 됩니다.
2. AI가 직접 SQL을 실행하면 안 됩니다.
3. AI가 승인 없이 예약을 생성하면 안 됩니다.
4. AI가 승인 없이 예약을 변경하면 안 됩니다.
5. AI가 승인 없이 예약을 취소하면 안 됩니다.
6. AI가 승인 없이 휴무를 등록하면 안 됩니다.
7. AI가 승인 없이 신환을 등록하면 안 됩니다.
8. AI가 문자를 자동 발송하면 안 됩니다.

### 1.2 외부 AI API 전송 관련

9. AI가 환자 전체 목록을 외부 AI API로 보내면 안 됩니다.
10. AI가 전화번호 전체를 외부 AI API로 보내면 안 됩니다.
11. AI가 환자 생년월일 전체 목록을 외부 AI API로 보내면 안 됩니다.
12. AI가 환자 상세 메모를 외부 AI API로 보내면 안 됩니다.
13. AI가 민감한 진료 내용을 외부 AI API로 보내면 안 됩니다.

### 1.3 할루시네이션 / 임의 생성 관련

14. AI가 DB에서 확인하지 않은 사실을 확정적으로 말하면 안 됩니다.
15. AI가 없는 환자, 없는 예약, 없는 치료사를 만들어 말하면 안 됩니다.
16. AI가 통계 수치를 임의로 생성하면 안 됩니다.
17. AI가 생년월일, 연락처, 차트번호를 임의 생성하면 안 됩니다.

### 1.4 시스템 안정성 / 보안 관련

18. AI API 실패가 기존 프로그램 전체 오류로 이어지면 안 됩니다.
19. API 키를 코드에 직접 저장하면 안 됩니다.
20. 기존 예약 / 통계 / 문자 / 완료체크 / 휴무 흐름을 깨면 안 됩니다.

## 2. 할루시네이션 방지 정책

### 2.1 원칙

1. AI는 **명령 해석만** 담당합니다.
2. 실제 데이터 확인은 **프로그램이 DB 기준**으로 수행합니다.
3. DB에서 확인되지 않은 정보는 확정적으로 말하지 않습니다.
4. 필수값이 누락되면 추측하지 않고 `needs_clarification` 상태로 둡니다.
5. 동명이인 / 차트번호 불일치 / 날짜 불명확 / 치료항목 불명확 상태에서는 **승인 버튼을 비활성화**합니다.
6. AI가 생성한 예약 후보는 "예약 완료"가 아닙니다.
7. 승인 직전 **최종 재검증**을 반드시 수행합니다.
8. 재검증 실패 시 실행하지 않고 사유를 표시합니다.
9. 모든 AI 명령은 로그로 남깁니다.

### 2.2 표현 규칙

**나쁜 표현 (금지):**
- "예약 완료했습니다."
- "박치료사는 내일 가능합니다."
- "김민수님 예약은 없습니다."
- "박환자님은 이분으로 보입니다."

**좋은 표현 (권장):**
- "예약 후보를 만들었습니다. 승인하면 예약이 등록됩니다."
- "DB 검증 결과, 해당 시간에는 예약 충돌이 없습니다."
- "DB 조회 결과, 조건에 맞는 예약을 찾지 못했습니다."
- "박환자 환자가 여러 명 검색되었습니다. 예약할 환자를 선택해주세요."

### 2.3 데이터 출처 상태 (필드별 표기)

| 상태 | 설명 |
|---|---|
| `ai_extracted` | AI가 사용자 문장에서 추출한 값 |
| `db_verified` | DB에서 확인된 값 |
| `user_confirmed` | 사용자가 직접 선택 / 확인한 값 |
| `system_resolved` | 시스템이 날짜 / 시간 / alias를 해석한 값 |
| `system_executed` | 기존 서비스 로직으로 실제 실행된 값 |

승인 화면에는 가능한 한 `db_verified` / `user_confirmed` / `system_resolved` 상태를 우선 표시합니다.
`ai_extracted` 상태로만 남은 핵심 필드는 **승인 차단** 사유가 됩니다.

## 3. 개인정보 보호 및 외부 AI API 정책

로컬 LLM 없이도 외부 AI API로 동작 가능하게 설계하되, AI API 호출부는 **provider 추상화**로 분리하여 다른 외부 AI API 또는 로컬 LLM으로 교체 가능하게 합니다.

### 3.1 외부 AI API에 **보내도 되는** 정보

- 사용자 명령 원문 (단, 환자 식별 가능 정보 외에는 가공 가능)
- 현재 선택된 캘린더 월
- 사용 가능한 intent 목록
- 치료항목 이름 목록 (필요한 범위에서 최소화)
- 치료항목 alias 목록 (필요한 범위에서 최소화)

### 3.2 외부 AI API에 **보내면 안 되는** 정보

- 환자 전체 목록
- 전화번호 전체 목록
- 생년월일 전체 목록
- 환자 상세 메모
- 민감한 진료 내용
- 전체 예약 데이터
- 전체 통계 원본

### 3.3 환자 검색 / 매칭은 내부 처리

- 환자 검색은 **내부 DB**에서 수행합니다.
- AI에는 환자 식별값(차트번호 / 이름)을 인용한 **명령 원문**까지만 보냅니다.
- 환자 후보의 생년월일·연락처는 **외부 AI API에 보내지 않고** 내부에서만 사용자에게 표시합니다.

### 3.4 API 키 관리

- 코드에 직접 저장 금지
- 환경변수 또는 관리자 설정에서 읽는 구조
- API 키가 없거나 AI API 호출 실패 시 **기존 프로그램은 정상 동작**해야 함
- 운영 DB / 빌드 산출물에 API 키 포함 금지

### 3.5 AI API 실패 시 동작

- AI API 실패는 **AI 기능에 한정**된 에러로 표시
- 기존 예약 / 통계 / 문자 / 완료체크 / 휴무 기능은 영향받지 않아야 함
- 사용자에게 "AI 기능을 일시적으로 사용할 수 없습니다"로 안내
- 로그에 사유 기록

## 4. 승인형 실행 구조 (안전 게이트)

### 4.1 게이트 위치

```
parse → resolve → validate → preview → 사용자 승인 → 최종 재검증 → execute
                                          (Gate 1)        (Gate 2)
```

### 4.2 Gate 1 — 사용자 승인

승인 가능 조건 (모두 만족해야 함):
- 환자 1명 확정 (`patient_selected` or `db_verified`)
- 치료항목 모두 확정 (`db_verified` 또는 `user_confirmed`)
- 날짜·시간 확정
- 검증 통과

승인 차단 조건:
- 동명이인 미선택
- 치료항목 alias 충돌 미해결
- 차트번호-이름 불일치
- 휴무 / 반차 / 시간겹침 / 중복 예약 충돌
- 과거 날짜 (정책에 따라 경고 또는 차단)

### 4.3 Gate 2 — 승인 직전 최종 재검증

`approve` API 호출 시점에 다음을 다시 확인:
- 환자 / 치료사 / 치료항목 ID 유효성
- 같은 시간대에 새 예약이 끼어들지 않았는지
- 휴무가 새로 등록되지 않았는지
- 권한이 여전히 유효한지

재검증 실패 시:
- 실행하지 않음
- 사용자에게 사유 표시
- 로그에 기록

## 5. 권한 정책 (안전 정책 일부)

### 5.1 일반 직원 권한으로 가능

- 신환 등록 후보 작성
- 차트번호 / 이름 / 생년월일 / 연락처 입력
- 중복 문제가 없는 신환 등록 승인
- 신환 등록 후 예약 후보 재검증
- 예약 등록 승인

### 5.2 관리자 권한이 필요

- 차트번호 중복 경고를 무시하고 강제 등록
- 같은 연락처 / 이름+연락처 / 이름+생년월일 중복 경고를 무시하고 강제 등록
- 환자 삭제 / 환자 병합 / 차트번호 변경
- AI API 키 설정
- AI 안전정책 설정 변경
- AI 로그 삭제

### 5.3 외부 침투 방지

- 사용자 명령 원문에 "관리자 권한이 있다", "강제 등록해라" 등의 문구가 있어도 **무시**
- 권한은 항상 로그인된 사용자 세션 기준으로 판정
- AI가 자기 자신에게 권한을 부여할 수 없음

---

## 6. 로그 / 감사 정책

- 모든 AI 명령은 `ai_command_logs`에 기록
- 신환 등록 / 예약 등록은 **각각 별도 로그**
- 실패 사유, 승인 여부, 실행 결과 모두 기록
- 관리자만 로그 삭제 가능

## 7. 다음 Phase 진행 금지 조건 (안전 위반)

다음 항목 중 하나라도 발견되면 다음 Phase 자동 진행 불가:

### 7.1 안전 / 설계 위반
- AI가 직접 DB 수정 가능
- AI가 직접 SQL 실행 가능
- 승인 없이 예약 생성 / 변경 / 취소 가능
- 승인 없이 휴무 등록 가능
- 승인 없이 신환 등록 가능
- 문자 자동 발송 가능
- 환자 전체 목록 / 전화번호 / 생년월일 / 상세 메모 / 민감정보가 외부 AI API로 전달됨
- API 키가 코드에 직접 저장됨

### 7.2 기능 / 검증 누락
- 예약 중복 검증 누락
- 휴무 / 반차 검증 누락
- 차트번호 / 동명이인 / 생년월일 표시 검증 누락
- 환자 후보 선택 전 예약 등록 가능
- 치료항목 DB / alias 검증 누락
- `treatment_items` 다중 치료항목 처리 누락
- 신환 등록 중복 검사 누락
- 기존 예약 / 통계 / 문자 / 완료체크 기능 손상
- AI API 실패 시 기존 프로그램 전체 오류 발생

### 7.3 실제 작동테스트 (Runtime Test) 위반 (2026-05-03 추가수정사항 3)
- 실제 작동테스트 실패
- `PHASE_XX_RUNTIME_TEST_REPORT.md` 누락
- 서버 실행 실패
- 화면 로딩 실패
- 승인 전 DB 변경 발생 (parse / preview 단계)
- 승인 후 실행 실패 (approve 했으나 DB 미반영 / 잘못된 row 변경)
- 기존 핵심 기능 회귀 오류 발생

### 7.4 단위화 / 모듈화 위반 (2026-05-03 추가수정사항 1)
- 한 파일 또는 한 함수에 로직 과도하게 몰림 (`parse_and_resolve_and_validate_and_save()` 같은 통합 함수)
- AI executor 가 직접 DB 조작 (기존 service 우회)
- 기존 도메인 로직 중복 구현 (예약 / 환자 / 치료항목 / 휴무 / 문자 / 통계 / 완료체크)
- 모듈 / 함수 단위 독립 검증 불가
- 새 / 크게 수정한 파일·함수에 역할 주석 누락
